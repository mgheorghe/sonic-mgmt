# Cisco SmartSwitch NPU config snapshots (keysight-css01 / MtFujiv2)

Saved to support reverting/merging config across SONiC image versions on the
Cisco 8102-28FH-DPU-O NPU (mgmt **10.36.77.121**).

## Device identifiers (for recovery after a factory reset)
- **mgmt eth0 MAC: `40:f4:9f:33:f1:60`** — use this to find the new DHCP IP if the
  mgmt address changes after a factory-default reset.
- Static mgmt today: `10.36.77.121/22`, gateway `10.36.76.1`.
- switch base MAC: `24:d5:e4:32:4e:11`, hwsku `Cisco-8102-28FH-DPU-O`,
  platform `x86_64-8102_28fh_dpu_o-r0`, hostname `MtFujiv2`, subtype `SmartSwitch`.
- login `admin` / `password`.

## Folders
- `working_20251110.16/` — config pulled from the **known-good image `20251110.16`**,
  on which redis TCP + the gNMI server (`:50051`) come up healthy.
  - `config_db.json` — startup config (`/etc/sonic/config_db.json`)
  - `running_config_all.json` — `show runningconfiguration all`
- `newimg_internal.167426684/` — captured on the **new image `internal.167426684-dfd1f4ecaa`**:
  - `factory_config_db.json` — factory default generated by
    `sonic-cfggen -H -k Cisco-8102-28FH-DPU-O --preset l1` (NOT applied to the live box,
    so the mgmt IP was preserved). Minimal: only **3** tables
    (`DEVICE_METADATA`, `FLEX_COUNTER_TABLE`, `PORT`) — note it has **no `MGMT_INTERFACE`**,
    so applying it as-is would drop the static IP to DHCP.
  - `migrated_config_db.json` — the new image's current `/etc/sonic/config_db.json`
    (the old config migrated forward; ~identical to the working config). This is what
    the new image runs when the redis-TCP wedge reproduces.

## Table comparison
Working config has **75** top-level tables; new-image factory has **3**. Everything
SmartSwitch/DASH-relevant (GNMI, TELEMETRY, DPU/DPUS/VDPU/REMOTE_DPU, DASH_HA_*,
MGMT_INTERFACE, MID_PLANE_BRIDGE, FEATURE, VNET, VXLAN_TUNNEL, BGP_*, etc.) exists
only in the working config — the factory preset is a bare L1 base to merge onto.

## Why
On the new image `internal.167426684-dfd1f4ecaa`, the NPU redis TCP listener
(`127.0.0.1:6379`) wedges within ~10 min (unix socket stays fine), so `sonic-db-cli`
and the gNMI server startup hang and the DASH push test can't run. The working
image does not exhibit this. Plan: compare the new image's factory-default config
against this working config to isolate whether a migrated config key triggers the
wedge.
